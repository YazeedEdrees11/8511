# Eighty Five Eleven ‚Äî Frontend Redesign + Multimodal RAG

**Date:** 2026-05-08
**Client:** Eighty Five Eleven (8511) ‚Äî streetwear / sneakers / accessories retailer, Swefieh Village, Amman, Jordan
**Source site:** https://www.eightyfiveeleven.com (current Wix build)

## 1. Goal

Rebuild the 8511 storefront as a modern Next.js application with a Stitch-driven design system, and add a multimodal Retrieval-Augmented Generation (RAG) chat experience so visitors can ask natural-language questions about products, services, and the store and receive grounded answers with inline product imagery.

Scope of v1: local development demo only. No production deploy, no payment integration, no headless commerce migration. Catalog is built by scraping the existing Wix site.

## 2. Non-goals

- No tool-calling / multi-step agent loop. Pure RAG pipeline only.
- No bilingual (Arabic) support in v1. English only.
- No Wix migration / replacement of checkout.
- No Pinecone, Qdrant, or external vector DB. FAISS local index only.
- No CMS / admin UI for editing the catalog. Catalog is regenerated by re-running the ingest script.
- No authentication, accounts, or order history.

## 5. Data shapes

```ts
type Product = {
  slug: string;
  name: string;
  brand: "nike" | "adidas" | "supreme" | "hats";
  price?: string;
  sizes?: string[];
  image_url: string;     // Local path under /images/products/
  source_url: string;    // Original Wix product page
  description: string;
};

type KBChunk = {
  id: string;
  type: "service" | "about" | "store" | "contact";
  title: string;
  text: string;
};

type IndexMeta = {
  id: string;            // Stable id (product slug or kb id)
  kind: "product" | "kb";
  modality: "text" | "image";
  ref: string;           // Pointer back into products.json or kb.json
};
```

The FAISS index stores one vector per `IndexMeta` row. Products contribute two vectors (one for `description + name`, one for `image`), KB chunks contribute one text vector each.

## 6. Ingestion pipeline (`scripts/ingest.ts`)

1. Use Firecrawl to map `eightyfiveeleven.com`.
2. Scrape brand pages (`/nike`, `/adidas`, `/supreme`, `/caps`) and follow `product-page/*` links.
3. For each product page, extract: name, image URL, price (if present), description, brand from URL path.
4. Download product images into `public/images/products/<slug>.jpg`.
5. Download the official logo from the scraped Wix asset URL into `public/logo.png`.
6. Build hand-curated `kb.json` for services (8 services from `/services`), store info (Swefieh Village location, contact email, IG link), and about copy.
7. Write `data/products.json` and `data/kb.json`.

## 7. Embedding pipeline (`scripts/embed.ts`)

Built around LangChain's `Document` + `FaissStore` abstractions:

1. Load `products.json` and `kb.json`.
2. Build LangChain `Document[]`:
   - One text `Document` per product: `pageContent = "${name}. ${description}"`, `metadata = { kind: "product", slug, modality: "text" }`.
   - One image `Document` per product: `pageContent = ""`, `metadata = { kind: "product", slug, modality: "image", imagePath }` (the embeddings class reads `metadata.imagePath` when modality is `image`).
   - One text `Document` per KB chunk: `metadata = { kind: "kb", id, type }`.
3. Instantiate `VertexMultimodalEmbeddings` (custom class extending `Embeddings`; routes `embedQuery`/`embedDocuments` to text path and falls back to image path when `metadata.modality === "image"`).
4. `FaissStore.fromDocuments(docs, embeddings)` ‚Üí `store.save("data/faiss")`.

## 8. RAG pipeline (`app/api/chat/route.ts`)

A LangChain `RunnableSequence` composed at startup and reused per request:

```
RunnableSequence:
  1. { question, history } passthrough
  2. retriever = FaissStore.asRetriever({ k: 6, filter: dedupeBySlug })
  3. ChatPromptTemplate:
       system: "You are 8511's storefront assistant. Ground every answer in
                the provided context. If unsure, say so. When recommending
                products, emit <product slug=\"..\"/> tags so the UI can render
                product cards."
       human:  "Context:\n{context}\n\nHistory:\n{history}\n\nQuestion: {question}"
  4. ChatHuggingFace (Qwen2.5-Instruct, streaming=true)
  5. StringOutputParser ‚Üí ReadableStream
```

The route handler returns the stream as `text/event-stream`. The client incrementally parses `<product slug="..."/>` tags and replaces them with `<ProductCard>` components reading from `products.json`.

No tool calling. No multi-step agent. Single retrieval + single generation per turn. Conversation history is rendered into the prompt as prior `user`/`assistant` turns; retrieval re-runs against the latest user query only.

## 9. Frontend

Pages mirror the existing Wix IA so the redesign feels familiar to returning customers:

- `/` Home ‚Äî hero, brand strip (Nike / Adidas / Supreme / Hats), services teaser, store-locator footer
- `/shop` ‚Äî all products grid with brand filter
- `/shop/[brand]` ‚Äî brand-specific grid
- `/product/[slug]` ‚Äî image, name, price, sizes, description, "Ask the assistant about this" CTA that opens the widget pre-filled
- `/services` ‚Äî the 8 numbered services
- `/about` ‚Äî culture / mission copy from current site
- `/contact` ‚Äî store address, email, IG, Google Maps link
- `/chat` ‚Äî full-screen RAG chat

The 8511 logo (pulled from Wix assets at ingest time) anchors the global header on every page.

`<ChatWidget>` is mounted in `app/layout.tsx` and is visible on every route. It and `/chat` both render `<ChatPanel>`, which is the single source of conversation behaviour.

## 10. Design system (Stitch)

Single Stitch project named **8511 Storefront**. Tokens seeded from the current site:

- **Type:** Futura / Questrial pairing (display / body), Roboto fallback
- **Palette:** monochrome base (off-white background, near-black text), single accent reserved for CTAs and the chat widget bubble ‚Äî exact accent picked during the Stitch pass
- **Imagery:** large-format product photography on neutral backgrounds, generous whitespace
- **Voice:** terse, culture-forward, mirrors existing copy ("Advocates of the Sneaker Streetwear Culture")

Screens generated in Stitch, exported as Tailwind components into `components/ui/`:
Home, Shop grid, PDP, Services, About, Contact, `/chat`, `ChatWidget` popover, `ProductCard` (chat-embedded variant), Header, Footer.

## 12. Chunking strategy

The corpus is dominated by short, atomic records, so the default is **no chunking**.

| Source | Typical size | Strategy |
|---|---|---|
| Products (name + description + price + sizes) | 30‚Äì80 tokens | One `Document` per product. No splitter. Splitting would orphan price/size from the name and hurt retrieval grounding. |
| Services (8 blurbs) | 20‚Äì40 tokens each | One `Document` per service. No splitter. |
| About / store / contact | 30‚Äì100 tokens each | One `Document` per record. No splitter. |
| Blog posts (deferred ‚Äî only if ingested later) | 300‚Äì1500 tokens | `RecursiveCharacterTextSplitter({ chunkSize: 500, chunkOverlap: 50, separators: ["\n\n", "\n", ". ", " "] })`. |

Semantic chunking is intentionally not used. It's overhead (extra embedding calls, tunable threshold) on a corpus that is already atomic and single-topic per record.

## 13. Risks & open questions

- **Vertex auth on local-only dev:** developer must have a Google Cloud project with Vertex AI enabled and a service-account JSON. Document this in the README.
- **HF Inference rate limits:** Qwen2.5-72B may hit free-tier limits during demo. Fall back to Qwen2.5-7B-Instruct if so.
- **Wix scrape brittleness:** if Wix changes its DOM, ingest may break. Acceptable for a v1 demo; re-run when needed.
- **Image embedding cost:** Vertex multimodal embeddings are billed per image. Catalog is small (~tens of products) so cost is negligible, but cache embeddings in `meta.json` to avoid recomputation.
- **No moderation layer:** Qwen output is not filtered. Acceptable for an internal demo; flag before any public deploy.
